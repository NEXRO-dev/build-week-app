# Echly Product Plan Summary

出典: `/Users/ryo/Downloads/Echly_Product_Plan.pdf`

## 概要

Echlyは、夜にユーザーが音声で今日の振り返り、明日の予定、悩みを話すだけで、声と発話内容からコンディションを推定し、翌日のタスク整理、予定調整、連絡文の下書きまで行う自律型AI秘書。

プロダクトの中心価値は、疲れている時ほどタスク管理や予定調整ができなくなる問題に対して、ユーザーの「働ける量」に合わせて仕事量を整えることにある。

北極星となるメッセージ:

> 疲れている時ほど、何も整理しなくてよい。話すだけで、明日が軽くなる。

## 解決する課題

- タスク管理ツールは入力、分類、更新の負荷が高く、疲れているほど使われにくい。
- ユーザーは疲労やストレスを過小評価し、翌日の予定を詰め込み続ける。
- ヘルスケアアプリは状態を可視化しても、実際の会議や仕事量までは減らしてくれない。
- 結果として、心身の状態と仕事量が切り離されたまま運用されている。

## Echlyの基本体験

1. 寝る前に30秒から2分ほど音声で話す。
2. 音声を文字起こしし、タスク、期限、相手、悩みを抽出する。
3. 声のエネルギー、テンポ、無音、声質などから負荷シグナルを推定する。
4. Google Calendarの予定やタスク情報と統合し、翌日の実行プランを作る。
5. 高負荷の場合は休息ブロック、延期候補、調整メール文面を提案する。
6. ユーザー承認後にCalendarへ反映し、Gmailに下書きを作成する。

## 目的と非目的

### 目的

- 入力負荷を最小化し、毎日の振り返りと計画を継続しやすくする。
- コンディションを仕事量の意思決定に反映する。
- 予定調整や連絡の心理的負担をAIが肩代わりする。
- 限界を迎える前に、予定の余白を作る。

### 非目的

- 精神疾患、疲労、ストレスの医学的診断。
- 本人の許可なしに会議を削除したり、メールを送信したりすること。
- 企業が個人の音声や会話内容を監視すること。
- 生産性スコアで社員を評価、順位付けすること。

負荷の表現は「ストレス85%」のような断定ではなく、「音声から推定した負荷シグナル」「通常時との差」として扱う。

## ターゲットユーザー

- 会議と締切が多い知識労働者
- 他者調整が多いマネージャー
- 短期集中が続く学生や開発者

主要なJobs to Be Done:

- 疲れて帰った時、考えを整理せずに明日の仕事を預けたい。
- 予定が過密な時、何を守り何を動かすべきか判断してほしい。
- 会議を動かす必要がある時、失礼のない連絡文をすぐ用意したい。
- 自分の調子の変化を、日々の予定との関係で理解したい。

| 機能 | 要件 | 優先度 |

| 音声チェックイン | ブラウザまたはデスクトップで録音、再生、送信 | Must |
| 文字起こし | 日本語音声をテキスト化 | Must |
| タスク抽出 | 予定、期限、相手、悩みをJSON化 | Must |

| Gmail連携 | 日程調整メールを下書き保存 | Must |
| 承認UI | 外部変更前に差分を確認 | Must |
| 履歴 | 過去7日分のチェックイン表示 | Should |

## 画面構成

- Home / Check-in: 大きな録音ボタン、今日の日付、前回の状態、すぐ始められる導線。
- Analysis: 波形、文字起こし、抽出タスク、推定負荷シグナル。
- Tomorrow Plan: 維持する予定、延期候補、集中時間、休息ブロックのタイムライン。
- Approval: 変更前後の差分、カレンダー操作、メール下書きの個別承認。
- History: 日別の負荷シグナル、予定密度、実行した調整、自己評価。
- Settings: Google連携、データ保存、音声削除、実行権限、安全設定。

デモでは、録音、解析のWow感、AIの判断理由、Google CalendarとGmailへの反映証拠を優先する。

## AIエージェント判断仕様

### 入力

- 文字起こしテキスト
- 音響特徴量と推定負荷レベル
- Google Calendarの翌日予定
- ユーザー設定の勤務時間、休息方針
- タスクの期限、重要度、関係者、移動可能性

### 判断原則

- 期限が近いもの、本人承認が必須な予定を優先して守る。
- 移動可能で重要度の低い予定から延期候補にする。
- 高負荷時は連続会議を避け、回復時間を入れる。
- 変更理由と影響を説明し、ユーザーが修正できるようにする。
- 外部への送信や削除は、明示承認なしに行わない。

### 出力イメージ

```json
{
  "condition": {},
  "extracted_tasks": [],
  "keep": [],
  "reschedule": [],
  "rest_blocks": [],
  "email_drafts": [],
  "rationale": []
}
```

## 負荷推定のMVP方針

利用する音響特徴:

- 音量、エネルギー
- 基本周波数の中央値と変動
- 発話速度
- 無音区間の割合
- 声質特徴
- 本人の通常時ベースラインとの差

ハッカソンでは精度競争を狙わず、簡易スコアとデモ用キャリブレーションを採用する。UI上では医療診断ではなく、あくまで「負荷シグナル」として扱う。

## 推奨アーキテクチャ

| レイヤー | 推奨技術 | 役割 |
| --- | --- | --- |
| Client | Next.js / React / PWA | 録音、波形、承認UI、スマホ対応 |
| Desktop wrapper | Tauri | Macアプリ化、通知、メニューバー常駐 |
| API | FastAPI または Next.js API | 認証、音声処理、エージェント制御 |
| Audio | MediaRecorder + librosa等 | 録音、特徴量抽出、簡易負荷推定 |
| Speech-to-text | 音声認識API | 発話の文字起こし |
| Agent | LLM + Structured Output | タスク抽出、判断、説明、メール生成 |
| Integrations | Google OAuth / Calendar / Gmail | 予定操作と下書き保存 |
| Data | PostgreSQL / Supabase | ユーザー設定、履歴、承認ログ |
| Hosting | Vercel + Cloud service | 短期開発とデプロイ |

処理フロー:

1. クライアントで音声録音。
2. 音声をAPIへアップロード。
3. 文字起こしと音響特徴抽出を並列実行。
4. Calendarから翌日予定を取得。
5. LLMが構造化プランを生成。
6. 差分を画面表示。
7. 承認された操作だけGoogle APIへ実行。
8. 監査ログと結果を保存。

## プライバシー・安全設計

必須ガードレール:

- 音声の生データは初期設定で処理後削除し、保存は明示オプトインにする。
- 負荷推定値は診断ではなく推定であると常時表示する。
- 予定削除、メール送信、外部共有は承認制にする。
- MVPではメール送信ではなく下書き作成までに留める。
- AIの提案理由と参照情報を表示する。
- ユーザーが負荷判定を訂正できるようにする。
- 企業管理者には個人の会話内容、音声、具体的予定を開示しない。

避けるべき設計:

- 高い負荷スコアを人事評価に使う。
- 声だけで健康状態を断定する。
- 本人の知らないところで常時録音する。
- AIが勝手に重要会議を削除する。
- 不安を煽る赤色警告を常用する。

## 差別化

- Frictionless: 最も疲れている時でも話すだけで使える。
- Contextual: 声だけでなく、発話内容、期限、カレンダーを組み合わせる。
- Agentic: アドバイスで止まらず、実行可能な変更案と連絡文を作る。
- Protective: 生産性を上げるだけでなく、働きすぎを抑える。

現実的なモート:

- 本人の通常時と比較する個人ベースライン。
- どの調整が翌日の自己評価改善につながったかの結果学習。
- Calendar、Gmail、チーム運用への深い統合。
- 透明性、承認制、プライバシー設計による信頼資産。

## ハッカソンデモ仕様

3分デモ構成:

1. 0:00-0:25 課題提示: 疲れている時ほどタスク管理はできない。
2. 0:25-0:55 音声入力: 重い予定と疲れを話す。
3. 0:55-1:20 解析: 波形、文字起こし、負荷シグナルを表示。
4. 1:20-1:55 提案: 維持、延期、休息ブロックを説明。
5. 1:55-2:30 実行: Calendar更新、Gmail下書き作成。
6. 2:30-3:00 Vision: 働ける量をAIが守るインフラへ。

成功条件:

- 録音開始から結果表示まで30秒以内。
- 予定変更はテスト用Googleアカウントで確実に成功。
- メールは送信せず下書き作成まで。
- ネットワーク障害時の録画バックアップを用意。
- 負荷値は固定モックではなく、最低1つの音響特徴に反応する。

チーム分担:

- Frontend: 録音、波形、プラン差分、承認UI
- Backend: 音声処理、DB、Google OAuth/API
- AI: 抽出スキーマ、判断プロンプト、評価ケース
- Design/Pitch: ストーリー、デモ演出、ブランド

最優先で勝つべき導線は、音声を話す、状態を理解する、予定を守るまたは減らす、Google上で結果が見える、という一連の体験。

## 検証計画とKPI

MVPで検証する仮説:

- ユーザーは夜の音声チェックインを手入力より楽だと感じる。
- AIの予定調整案は、自分では決めづらい仕事を減らす。
- 状態推定が完璧でなくても、仕事量を見直すきっかけとして価値がある。
- 外部変更前の承認があれば、AIエージェントへの信頼を維持できる。

| 指標 | 意味 | 初期目標 |
| --- | --- | --- |
| チェックイン完了率 | 録音開始からプラン確認まで | 70%以上 |
| 平均入力時間 | 利用負荷 | 2分未満 |
| 提案承認率 | 判断の有用性 | 40%以上 |
| 翌日自己評価 | 調整後の体感 | 改善傾向 |
| 誤判定訂正率 | 負荷推定の妥当性 | 継続的に低下 |
| 継続率 | 習慣化 | 週3回以上 |

## 未決事項

- 最初の提供形態をPWAのみとするか、TauriでMacアプリ化するか。
- 負荷表示を数値、3段階、通常時との差のどれにするか。
- Calendar変更を自動適用、常に承認、信頼度別のどれにするか。
- 音声データは即削除を標準とするか。
- 初期ユーザーを個人開発者、会社員、学生のどこに絞るか。
- B2B展開時のデータ境界と管理者権限をどう設計するか。
- 効果検証の主要結果を自己評価、睡眠、予定密度のどれにするか。

推奨する初期決定:

- 提供形態: スマホ対応PWA。デモPCはTauriラップも可。
- 表示: 「負荷シグナル 高 / 通常より高い」。数値は補助。
- 権限: Calendar変更とメール下書きは必ず確認画面を通す。
- ターゲット: 会議と締切の多い知識労働者。
- 価値訴求: 「メンタル診断」ではなく「明日の仕事量を守る」。
